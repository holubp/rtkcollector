# UM980 Complete E2E V1 Design

## Status

Approved design for the next implementation cycle. This design completes the
practical end-to-end V1 implementation for Unicore UM980 / Unicore N4 receivers
only. Other receiver profiles remain modelled but are not practical E2E targets
in this cycle.

## Goals

- Make UM980 rover recording operational end to end.
- Make UM980 rover + external NTRIP corrections operational end to end.
- Make UM980 temporary-base preparation operational end to end for external
  `um980-rtklib-pipeline` processing.
- Enable UM980 PPP solution recording for temporary-base and base-preparation
  workflows, with a recommended minimum occupation time of 15 minutes.
- Make fixed-base operation operational from either manual coordinates or an
  imported `base-position.json`.
- Support fixed-base RTCM output as local recording and local RTCM extraction.
- Provide a practical live quality dashboard for rover and base sessions.
- Preserve byte-exact raw recording as the authoritative data path.

## Non-Goals

- No Android-side PPP/static solver.
- No Android-side RTKLIB real-time solution.
- No NTRIP caster/server upload from Android in this cycle.
- No rtk2go upload preset in this cycle.
- No maps, shapefiles, GIS editing, field-feature collection or cartographic
  survey project management.
- No persistent UM980 receiver changes generated by the app.
- No practical E2E support for u-blox or generic receivers in this cycle.

## Workflows In Scope

### Plain Rover Recording

The app configures the UM980 as a rover, records the receiver byte stream,
records app-sent commands, parses live solution/quality where available and
keeps recording independent of parser failures.

### Rover With External NTRIP To UM980

Android acts as an NTRIP client for an external base source such as EUREF or a
CORS caster. RTCM bytes received from NTRIP are written to:

- `correction-input.raw` before injection; and
- `tx-to-receiver.raw` before being transmitted to the UM980 serial input.

The receiver internal solution remains the primary live rover solution. The app
does not run RTKLIB in V1.

### Temporary-Base Preparation

The receiver is stationary. The app records raw observations at at least 1 Hz
where supported, records the in-device solution and enables UM980 PPP
capability/output. The session is intended to be processed later by the external
`um980-rtklib-pipeline` or another external static/PPP/static-RTK workflow.

The UI must communicate that 15 minutes is the practical minimum occupation time
for useful UM980 PPP-assisted base preparation, with longer sessions preferred
when possible.

### Temporary-Base Preparation With External NTRIP To UM980

The app may feed EUREF/CORS/NTRIP corrections to the stationary UM980 while
recording. This is still a recording/export workflow. It does not produce an
accepted base coordinate on Android.

### Fixed Base From Accepted Coordinate

The app configures the UM980 as a fixed base from a selected accepted coordinate
source. Supported coordinate sources:

- manual coordinate entry; and
- imported `base-position.json`.

The source of truth is explicit. If both manual fields and imported JSON exist,
the user must choose which source is active. The selected source is recorded in
session metadata.

### Fixed Base With Local RTCM Output

The app provides a visible fixed-base RTCM-output preset. UM980 RTCM output is
recorded in the receiver RX stream and extracted locally into
`rtcm-extracted.rtcm3`. Android does not upload the base RTCM stream to a caster
in this cycle.

## Architecture

The implementation remains service-first:

1. UI builds a validated `WorkflowSpec` and UM980 runtime profile.
2. The foreground service executes the validated workflow.
3. USB read bytes are written to `receiver-rx.raw` before any parser,
   extractor, monitor or UI consumer sees them.
4. Advisory consumers parse solution, PPP status and RTCM frames from copied RX
   bytes.
5. NTRIP client bytes are recorded before receiver injection.
6. App TX to receiver is recorded before write.
7. Session metadata is written separately and never contains raw secrets.

Parser, RTCM extraction, quality monitoring, NTRIP client and UI failures must
not stop or mutate raw recording while the USB transport and storage path still
function.

## UM980 Command Profiles

Generated commands are runtime-only. The app must not generate:

- `SAVECONFIG`;
- receiver reset commands;
- firmware/update commands;
- erase, format or factory-reset commands;
- commands that persistently mutate receiver state.

All generated command plans are visible and editable before start. Edited
commands go through the same safety validator as generated commands.

### Common Init

Common init should:

- optionally apply the selected profile baud/runtime baud sequence;
- clear unknown previous output with `UNLOG COM1`;
- configure output needed for the selected workflow;
- avoid persistent changes.

### Rover Mode

Rover mode should:

- set `MODE ROVER`;
- request raw observations where supported;
- request GGA and UM980-native solution output such as `BESTNAVB`;
- optionally accept external RTCM from Android NTRIP client injection.

### Temporary-Base Mode

Temporary-base mode should:

- keep the receiver stationary but not configure fixed-base operation;
- request raw observations at at least 1 Hz where supported;
- request in-device solution output;
- enable UM980 PPP capability/output;
- request PPP solution/status logs where documented and stable.

### Fixed-Base Mode

Fixed-base mode should:

- require an accepted coordinate source before start;
- generate UM980 fixed-base coordinate commands from the selected source;
- offer a status-only preset and an RTCM-output preset;
- record all generated and user-edited commands to `tx-to-receiver.raw`.

### Command Verification Source

Before implementation, exact UM980 command syntax for PPP enablement, PPP
solution/status logs, fixed-base coordinate configuration and RTCM output must
be verified against the local Unicore N4 command PDF and the adjacent
`um980-rtklib-pipeline` evidence. The implementation must cite the checked
manual sections in code comments or developer documentation where the syntax is
not self-evident.

## Session Artifacts

Every real session creates a folder with:

- `session.json`;
- `receiver-rx.raw`;
- `tx-to-receiver.raw`;
- `correction-input.raw` when external corrections are configured;
- `events.jsonl`;
- `quality-live.jsonl`;
- `receiver-solution.jsonl`;
- `receiver-ppp-solution.jsonl` when PPP is enabled/available;
- `base-position.json` for fixed-base sessions;
- `rtcm-extracted.rtcm3` for fixed-base RTCM-output sessions.

`session.json` must include:

- workflow id and name;
- receiver role;
- receiver profile id;
- selected UM980 profile id;
- coordinate source for fixed-base workflows;
- base-position metadata when applicable;
- NTRIP caster/mountpoint metadata without passwords or tokens;
- secret presence or secret reference for stored NTRIP credentials;
- validation result at session start;
- artifact expectations;
- start and stop timestamps;
- Android device and USB metadata.

## Live Monitoring

The recording monitor must include:

- running state and elapsed time;
- session path;
- receiver RX byte count and throughput;
- app TX-to-receiver byte count;
- correction input byte count and throughput;
- NTRIP client state, reconnect count and correction age;
- parsed GGA solution state;
- parsed BESTNAV/BESTNAVB solution state where available;
- PPP status/solution for temporary-base workflows where available;
- RTCM frame counts by message type;
- raw observation presence/rate;
- warning history for parser, command, NTRIP, storage, USB and quality issues;
- base-specific warnings for short PPP occupation time, missing antenna
  metadata and missing frame/epoch.

Monitoring is advisory. Dropped monitor samples are acceptable if required to
protect raw capture.

## UI Design

The current single advanced screen becomes a guided workflow with an advanced
command editor.

Pre-start flow:

1. Choose workflow.
2. Choose UM980 receiver, USB device and baud settings.
3. Configure workflow context:
   - rover NTRIP caster/mountpoint/GGA upload;
   - temporary-base PPP occupation target and antenna metadata;
   - fixed-base coordinate source;
   - fixed-base RTCM output preset.
4. Preview generated UM980 commands.
5. Optionally edit init, mode and shutdown command sequences.
6. Validate and start.

Recording monitor:

- always-visible stop action;
- session and byte counters;
- solution/PPP/RTCM/quality panels;
- warning history;
- clear distinction between receiver internal solution, PPP solution and
  correction stream state.

Responsive requirements:

- phone portrait: single-column wizard and monitor;
- phone landscape: navigation/summary on the left, active step or monitor on
  the right;
- tablet/wide: persistent workflow sidebar, main panel and right status/warnings
  panel;
- rotation preserves form state and reconnects to running service state.

## Secrets And Profiles

V1 persists NTRIP client credentials only. Credentials are stored using
Android-keystore-backed storage. Session exports contain redacted metadata and
secret presence/reference only.

Non-secret profile data may be persisted:

- UM980 command profiles;
- selected workflow defaults;
- NTRIP host/port/mountpoint;
- GGA upload preference;
- base coordinate entries;
- antenna metadata;
- UI defaults.

## Testing Plan

Unit tests:

- UM980 command generation for rover, temporary-base PPP and fixed-base modes.
- UM980 command safety denial for persistent/risky commands.
- Base-position JSON import/export and coordinate-source selection.
- Session metadata redaction and artifact expectations.
- NTRIP client request/redaction and reconnect state.
- NMEA GGA parsing.
- BESTNAV/BESTNAVB parsing.
- PPP solution/status parsing where the documented message format is
  implemented.
- RTCM3 frame extraction and CRC.
- Capture runtime proving raw RX write happens before advisory consumers and
  advisory failures do not stop recording.
- Workflow/UI state validation.

Build validation:

- On Windows or another full Android build host:
  - `.\gradlew clean`
  - `.\gradlew test`
  - `.\gradlew assembleDebug`

Manual Android test matrix:

- phone portrait;
- phone landscape;
- tablet portrait;
- tablet or wide landscape;
- rover without NTRIP;
- rover with EUREF/CORS NTRIP to UM980;
- temporary-base recording with PPP enabled for at least 15 minutes;
- temporary-base with NTRIP to UM980;
- fixed base from manual coordinate;
- fixed base from imported `base-position.json`;
- fixed base RTCM output with local RTCM extraction;
- stop/shutdown while recording;
- screen-off/background recording.

## Risks

- Wrong fixed-base coordinates can produce precise-looking but wrong rover
  positions. UI and session metadata must make coordinate source, frame, epoch
  and antenna metadata visible.
- UM980 PPP command syntax must be verified before implementation; incorrect
  commands may silently produce no PPP output.
- Full RTCM extraction and parsing must be isolated so malformed bytes cannot
  affect raw capture.
- Android background restrictions may still stop the app after user force-stop,
  USB power loss, phone shutdown or aggressive vendor task killing.
- Responsive UI behaviour must be verified on real or emulated phone/tablet
  sizes; a compiling UI is not enough.

## Deferred Work

- Android NTRIP caster/server upload.
- rtk2go upload preset.
- Android-side static RTK, PPP or RTKLIB solution.
- u-blox and generic receiver practical E2E paths.
- Maps, GIS, shapefiles and feature collection remain out of scope.
